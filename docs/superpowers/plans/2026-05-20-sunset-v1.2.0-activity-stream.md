# Sunset v1.2.0 — Activity Stream Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Ship a realtime worker activity stream — new Activity page in the dashboard powered by Server-Sent Events backed by Redis pub/sub.

**Architecture:** ActivityRecorder listener → ActivityEvent value object → Redis sorted set (replay buffer, cap 5000) + Redis pub/sub channel. SSE endpoint fans pub/sub to connected browsers with Last-Event-ID resume + heartbeat + max-connection-seconds. Vue Activity.vue consumes via EventSource with auto-reconnect.

**Tech Stack:** PHP 8.2+, Laravel 10–12, Redis pub/sub via `Illuminate\Contracts\Redis\Factory`, Symfony StreamedResponse, Inertia + Vue 3 EventSource on the frontend.

**Spec:** `docs/superpowers/specs/2026-05-20-v1.2.0-activity-stream-design.md` — read it first.

### Task 4: RedisActivityRepository (read + write)

**Files:**
- Create: `src/Repositories/Redis/RedisActivityRepository.php`
- Create: `tests/Integration/Activity/RedisActivityRepositoryTest.php`
- Create: `tests/Unit/Activity/RedisActivityRepositoryFailureTest.php`

Implements `ActivityRepository` (public contract) and adds an internal `record(ActivityEvent $event): ActivityEvent` write method that:
1. INCR `sunset:activity:next_id` → real id
2. Builds a new `ActivityEvent` with the real id (since the factory passed 0)
3. ZADD `sunset:activity:stream` with score=id, member=`"{id}:{json}"`
4. ZREMRANGEBYRANK to cap at `sunset.activity.stream_buffer_size`
5. PUBLISH `sunset:activity:pubsub` JSON
6. Returns the assigned-id `ActivityEvent`

Read methods:
- `recent($limit)` — `ZREVRANGE sunset:activity:stream 0 limit-1`, parse, return as `list<ActivityEvent>` (descending id)
- `since($afterId, $limit)` — `ZRANGEBYSCORE sunset:activity:stream ($afterId +inf LIMIT 0 $limit`, parse, return ascending
- `before($beforeId, $limit)` — `ZREVRANGEBYSCORE sunset:activity:stream ($beforeId -inf LIMIT 0 $limit`, parse, return descending

All three read methods return `[]` if Redis throws (try/catch + return empty). `record()` does NOT swallow — that's the recorder's job.

### Task 6: ActivityStreamer (SSE generator)

**Files:**
- Create: `src/Activity/ActivityStreamer.php`
- Create: `tests/Unit/Activity/ActivityStreamerTest.php`

The streamer owns the SSE loop. Pure-ish — takes injected closures for sleep, time, pub/sub iterator, and an output sink. Tests inject all four.

```php
class ActivityStreamer
{
    public function __construct(
        private ActivityRepository $repository,
        private RedisFactory $redis,
        private string $redisConnection,
        private string $channel,
        private int $maxConnectionSeconds,
        private int $heartbeatIntervalSeconds,
        private int $pubsubBlockSeconds,
        private \Closure $clock,            // fn(): float
        private \Closure $sleep,            // fn(int $seconds): void
        private \Closure $emit,             // fn(string $sseFrame): void
    ) {}

    public function stream(?int $lastEventId): void
    {
        $start = ($this->clock)();
        $lastHeartbeat = $start;

        // 1. Replay missed events from the sorted set.
        if ($lastEventId !== null && $lastEventId >= 0) {
            foreach ($this->repository->since($lastEventId, 1000) as $event) {
                ($this->emit)($this->frame($event));
                $lastEventId = $event->id;
            }
        }

        // 2. Pub/sub loop until max_connection_seconds.
        $client = $this->redis->connection($this->redisConnection);
        $client->subscribe([$this->channel], function ($message) use (&$lastHeartbeat, $start) {
            $event = ActivityEvent::fromJson($message);
            ($this->emit)($this->frame($event));
            $lastHeartbeat = ($this->clock)();

            if (($this->clock)() - $start >= $this->maxConnectionSeconds) {
                throw new StreamCompleteException();
            }
        });
    }

    private function frame(ActivityEvent $event): string
    {
        return sprintf("id: %d\nevent: %s\ndata: %s\n\n", $event->id, $event->type, $event->toJson());
    }
}
```

Reality check: phpredis/predis pub/sub `subscribe()` is blocking. Heartbeats need to fire on a timer separate from message arrival. The cleanest way is to set the Redis client's `read_timeout` to `pubsub_block_seconds`, catch the read timeout, emit a heartbeat, check elapsed time, and re-subscribe. **This is fiddly across phpredis vs predis** — see implementation note below.

**Implementation note:** Both predis and phpredis support a configurable read timeout. The recommended approach:
- For phpredis: `$client->setOption(\Redis::OPT_READ_TIMEOUT, $pubsubBlockSeconds)`. On timeout, phpredis returns false from the subscribe callback's internal read; loop, emit heartbeat, check elapsed.
- For predis: use `Client::loop()` or a custom pump. Predis is more flexible here.

If wrangling both is too much for one task, **fall back to a simpler model**: skip the pub/sub subscribe entirely. Instead, poll the sorted set every `pubsub_block_seconds`:

```php
while (($this->clock)() - $start < $this->maxConnectionSeconds) {
    $newEvents = $this->repository->since($lastEventId, 100);
    foreach ($newEvents as $event) {
        ($this->emit)($this->frame($event));
        $lastEventId = $event->id;
    }

    $now = ($this->clock)();
    if ($now - $lastHeartbeat >= $this->heartbeatIntervalSeconds) {
        ($this->emit)(":heartbeat\n\n");
        $lastHeartbeat = $now;
    }

    ($this->sleep)($this->pubsubBlockSeconds);
}
```

**Picking the simpler polling model is the right call for v1.2.0** — it works identically across phpredis/predis, is testable with injected clock+sleep, and "every 5 seconds" is already faster than the existing 3s page polling. We document that "stream" means "fast cursor poll, server-side" rather than true Redis pub/sub. Pub/sub becomes a v1.2.1 optimization if anyone complains. Rename the `pubsub_block_seconds` config key → `poll_interval_seconds` to match reality.

**Update the spec's "pub/sub channel" subsection to reflect this decision** before writing code. Edit `docs/superpowers/specs/2026-05-20-v1.2.0-activity-stream-design.md` to replace the pub/sub mentions with "fast server-side polling of the sorted set" and update Task 1's config block + Task 4's record() (remove the PUBLISH step) accordingly.
