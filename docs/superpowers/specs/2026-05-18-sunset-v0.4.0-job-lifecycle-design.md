# Sunset v0.4.0 Design Spec — Job Lifecycle Subsystem

**Date:** 2026-05-18
**Status:** Approved for implementation planning
**Predecessor:** v0.3.0 (Redis transport) — merged 2026-05-18

## Goal

Move the queueing-and-recording subsystem out of `laravel/horizon` and into the
`Admnio\Sunset` namespace. After v0.4.0, no class under `src/Transports/` or
`src/Listeners/` references `Laravel\Horizon\Events\*`, `Laravel\Horizon\Listeners\*`,
or `Laravel\Horizon\JobPayload`. Sunset owns the events, the listeners, the
repositories, the payload type, and the Redis keyspace they write to. Horizon's
dashboard keeps working unchanged by way of thin adapter classes bound to
Horizon's repository contracts.

## Non-goals (deferred to later releases)

- **`sunset:work` supervisor** — replaces `php artisan horizon`. Deferred to **v0.5.0**.
- **Sunset SPA dashboard + dropping `laravel/horizon` from composer** — deferred to **v1.0.0**.
- **Worker CPU/memory monitoring** — v1.1.0.
- **Realtime worker activity stream** — v1.2.0.
- **Queue pause/resume controls** — v1.3.0.

v0.4.0 ships strictly the job-lifecycle subsystem. `laravel/horizon` stays as
a composer dependency (the dashboard, the supervisor binaries, and the
`ProcessRepository` / `SupervisorRepository` / `MasterSupervisorRepository` /
`HorizonCommandQueue` contracts all stay with Horizon).

## Decisions locked during brainstorming

1. **Scope decomposition.** The full Horizon-unbrand effort is split into three
   releases:
   - v0.4.0 — job lifecycle (this spec).
   - v0.5.0 — `sunset:work` supervisor.
   - v1.0.0 — Sunset dashboard + drop `laravel/horizon` dependency.

2. **Redis keyspace.** Rename `horizon:*` → `sunset:*` in v0.4.0. A
   `sunset:migrate-horizon-keys` artisan command provides an idempotent
   migration for existing installs.

3. **Repository interface design.** Define our own contracts under
   `Admnio\Sunset\Contracts\`. Thin adapter classes implement Horizon's
   repository contracts and delegate to ours, so `app(Laravel\Horizon\
   Contracts\JobRepository::class)` (used by the dashboard) transparently
   serves data from the sunset keyspace.

4. **Listener surface in our namespace.** Reimplement the ~9 Horizon lifecycle
   listeners in `Admnio\Sunset\Listeners\`. After v0.4.0 no
   `Laravel\Horizon\Listeners\*` class appears in the runtime listener chain
   for any lifecycle event.

5. **Event names.** Follow Laravel's past-tense convention
   (`JobQueueing`, `JobQueued`, `JobReserved`, `JobReleased`, `JobCompleted`,
   `JobFailed`) rather than Horizon's `JobPending`/`JobPushed`.

6. **MetricsRepository included.** Splitting out metrics avoids a keyspace
   split — without it, lifecycle data lives in `sunset:metrics:*` won't exist
   while Horizon's metrics still write to `horizon:metrics:*`. Including
   `MetricsRepository` in v0.4.0 keeps the keyspace consistent.

7. **ProcessRepository stays with Horizon.** Process state belongs to the
   supervisor work in v0.5.0.

## Architecture

```
Transports (SqsQueue, RedisQueue)
  │ push / pop / release
  ▼
Sunset events (Admnio\Sunset\Events\Job{Queueing,Queued,Reserved,
                                       Released,Completed,Failed})
  │
  ▼
Sunset listeners (registered via SunsetServiceProvider::boot)
  - StorePendingJob, StoreJob, MarkJobAsReserved, MarkJobAsReleased,
    MarkJobAsComplete, MarkJobAsFailed, MonitorTag,
    TranslateJobProcessed, TranslateJobFailed
  │
  ▼
Sunset repositories (Redis impls; our own contracts)
  - RedisJobRepository, RedisFailedJobRepository,
    RedisTagRepository, RedisMetricsRepository
  │
  ▼
Redis keyspace: sunset:*

— Dashboard read path —
Horizon dashboard
  → app(Laravel\Horizon\Contracts\JobRepository::class)
  → HorizonJobRepositoryAdapter (impls Horizon contract)
  → Admnio\Sunset\Contracts\JobRepository
  → reads from sunset:*
```

### Boundary rules

- **`src/Transports/` boundary.** No `use Laravel\Horizon\*` imports. Transport
  queues import `Admnio\Sunset\Events\*` and `Admnio\Sunset\JobPayload`.
- **`src/Listeners/` boundary.** No `use Laravel\Horizon\*` imports.
- **`src/Repositories/` boundary.** No `use Laravel\Horizon\*` imports.
- **`src/Adapters/Horizon/` boundary.** This is the *only* place that imports
  `Laravel\Horizon\Contracts\*` and `Laravel\Horizon\JobPayload`. The
  rationale for the adapter directory's name and location is to make this
  boundary explicit: when v1.0.0 drops the Horizon dep, the entire
  `Adapters/Horizon/` tree gets deleted.

## Events

All six concrete events extend `Admnio\Sunset\Events\JobEvent` (abstract).
Constructor: `(string $connectionName, string $queue, JobPayload $payload)`.
Public readonly props: `$connectionName`, `$queue`, `$payload`.

| Event           | Fired when                                                                                                | Dispatched by                                            |
|-----------------|-----------------------------------------------------------------------------------------------------------|----------------------------------------------------------|
| `JobQueueing`   | Right before transport hands payload to underlying queue. Listeners may mutate payload (tag injection).   | `RedisQueue::pushRaw`, `SqsQueue::pushRaw`               |
| `JobQueued`     | After underlying queue acks the push.                                                                     | Same                                                     |
| `JobReserved`   | A worker just popped the job from the queue.                                                              | `RedisQueue::pop`, `SqsQueue::pop`                       |
| `JobReleased`   | Worker requeued the job with a delay (e.g., manual release or transient failure).                         | Existing Laravel `$job->release()` instrumentation hook  |
| `JobCompleted`  | Worker finished successfully.                                                                             | `TranslateJobProcessed` listener on Laravel's event      |
| `JobFailed`     | Worker failed after exhausting retries.                                                                   | `TranslateJobFailed` listener on Laravel's event         |

**Why `JobCompleted`/`JobFailed` are translated, not dispatched directly by
transports:** the worker process — not the transport — is what detects
success/failure. `Illuminate\Queue\Events\JobProcessed` and `JobFailed` are
Laravel stock events emitted by `Illuminate\Queue\Worker`. The two tiny
translator listeners re-dispatch them as Sunset events. After translation, the
entire downstream listener chain matches on Sunset events only — no mixed
Laravel/Sunset event types in the recording layer.

## Listeners

One file per listener under `src/Listeners/`. Each has a single public method
`handle(EventClass $event): void`. Constructor takes the repos it needs.

| Listener                  | Listens on                                  | Writes                                                                                                                  |
|---------------------------|---------------------------------------------|-------------------------------------------------------------------------------------------------------------------------|
| `StorePendingJob`         | `JobQueueing`                               | `JobRepository::pushed()` (writes pending record before queue ack), `TagRepository::addTemporary()`                     |
| `StoreJob`                | `JobQueued`                                 | promotes pending → recent in `JobRepository`; reconciles tags                                                           |
| `MarkJobAsReserved`       | `JobReserved`                               | `JobRepository::reserved()`                                                                                             |
| `MarkJobAsReleased`       | `JobReleased`                               | `JobRepository::released()`                                                                                             |
| `MarkJobAsComplete`       | `JobCompleted`                              | `JobRepository::completed()`, `MetricsRepository::incrementThroughput()`                                                 |
| `MarkJobAsFailed`         | `JobFailed`                                 | `FailedJobRepository::failed()`, mirrors to `JobRepository::completed(silenced: false)` so the timeline stays consistent |
| `MonitorTag`              | `JobQueueing`                               | `TagRepository::monitor()` only when payload carries monitored tags                                                     |
| `TranslateJobProcessed`   | `Illuminate\Queue\Events\JobProcessed`      | re-dispatches as Sunset `JobCompleted`                                                                                  |
| `TranslateJobFailed`      | `Illuminate\Queue\Events\JobFailed`         | re-dispatches as Sunset `JobFailed`                                                                                     |

## Adapters

Adapter classes implement Horizon's repository contracts and delegate to our
contracts. They live in `src/Adapters/Horizon/`. The directory exists as a
single deletion target for v1.0.0 (when we drop the Horizon dep).

Three adapter classes:
- `HorizonJobRepositoryAdapter` — implements `Laravel\Horizon\Contracts\JobRepository`. Delegates lifecycle methods to our `JobRepository` and failure methods to our `FailedJobRepository` (Horizon's interface bundles them; ours splits them).
- `HorizonTagRepositoryAdapter` — implements `Laravel\Horizon\Contracts\TagRepository`. Delegates to our `TagRepository`.
- `HorizonMetricsRepositoryAdapter` — implements `Laravel\Horizon\Contracts\MetricsRepository`. Delegates to our `MetricsRepository`.

Adapters do `Laravel\Horizon\JobPayload` ↔ `Admnio\Sunset\JobPayload` wrapping
at every method that takes or returns a payload. Both classes share the same
internal shape (`->value` is the JSON string, `->id()`, `->decoded()`,
`->tags()` semantics), so wrapping is a `new` call away.

`HorizonRepositoryAdapter` (sketch — Job adapter shown):

```php
namespace Admnio\Sunset\Adapters\Horizon;

use Admnio\Sunset\Contracts\JobRepository as SunsetJobRepository;
use Admnio\Sunset\Contracts\FailedJobRepository as SunsetFailedJobRepository;
use Admnio\Sunset\JobPayload as SunsetJobPayload;
use Laravel\Horizon\Contracts\JobRepository as HorizonJobRepository;
use Laravel\Horizon\JobPayload as HorizonJobPayload;

class HorizonJobRepositoryAdapter implements HorizonJobRepository
{
    public function __construct(
        private SunsetJobRepository $jobs,
        private SunsetFailedJobRepository $failed,
    ) {}

    public function pushed($connection, $queue, HorizonJobPayload $payload)
    {
        $this->jobs->pushed($connection, $queue, $this->wrap($payload));
    }

    public function reserved($connection, $queue, HorizonJobPayload $payload)
    {
        $this->jobs->reserved($connection, $queue, $this->wrap($payload));
    }

    public function released($connection, $queue, HorizonJobPayload $payload, $delay = 0)
    {
        $this->jobs->released($connection, $queue, $this->wrap($payload), $delay);
    }

    public function completed(HorizonJobPayload $payload, $failed = false, $silenced = false)
    {
        $this->jobs->completed($this->wrap($payload), $silenced);
    }

    public function failed($exception, $connection, $queue, HorizonJobPayload $payload)
    {
        $this->failed->failed($exception, $connection, $queue, $this->wrap($payload));
    }

    public function findFailed($id) { return $this->failed->findFailed($id); }
    public function getFailed($afterIndex = null) { return $this->failed->getFailed($afterIndex); }
    public function countFailed() { return $this->failed->countFailed(); }
    public function totalFailed() { return $this->failed->totalFailed(); }
    public function countRecentlyFailed() { return $this->failed->countRecentlyFailed(); }
    public function deleteFailed($id) { return $this->failed->deleteFailed($id); }
    public function trimFailedJobs() { $this->failed->trimFailedJobs(); }

    // … remaining Horizon JobRepository methods delegate to $this->jobs …

    private function wrap(HorizonJobPayload $h): SunsetJobPayload
    {
        return new SunsetJobPayload($h->value);
    }
}
```

```php
$this->app->singleton(\Laravel\Horizon\Contracts\JobRepository::class, /* ... */);
$this->app->singleton(\Laravel\Horizon\Contracts\TagRepository::class, /* ... */);
```

```php
// Register Sunset event → listener map.
$events = $this->app['events'];
$events->listen(\Admnio\Sunset\Events\JobQueueing::class,  \Admnio\Sunset\Listeners\StorePendingJob::class);
$events->listen(\Admnio\Sunset\Events\JobQueueing::class,  \Admnio\Sunset\Listeners\MonitorTag::class);
$events->listen(\Admnio\Sunset\Events\JobQueued::class,    \Admnio\Sunset\Listeners\StoreJob::class);
$events->listen(\Admnio\Sunset\Events\JobReserved::class,  \Admnio\Sunset\Listeners\MarkJobAsReserved::class);
$events->listen(\Admnio\Sunset\Events\JobReleased::class,  \Admnio\Sunset\Listeners\MarkJobAsReleased::class);
$events->listen(\Admnio\Sunset\Events\JobCompleted::class, \Admnio\Sunset\Listeners\MarkJobAsComplete::class);
$events->listen(\Admnio\Sunset\Events\JobFailed::class,    \Admnio\Sunset\Listeners\MarkJobAsFailed::class);
$events->listen(\Illuminate\Queue\Events\JobProcessed::class, \Admnio\Sunset\Listeners\TranslateJobProcessed::class);
$events->listen(\Illuminate\Queue\Events\JobFailed::class,    \Admnio\Sunset\Listeners\TranslateJobFailed::class);

// Suppress Horizon's lifecycle listeners. Horizon's EventServiceProvider runs
// after ours (alphabetical: `L` > `A`), so we hook via app->booted to run
// AFTER Horizon's listeners are registered, then forget the ones we own and
// re-register ours.
$this->app->booted(function () use ($events) {
    $ourMap = [
        \Admnio\Sunset\Events\JobQueueing::class => [
            \Admnio\Sunset\Listeners\StorePendingJob::class,
            \Admnio\Sunset\Listeners\MonitorTag::class,
        ],
        \Admnio\Sunset\Events\JobQueued::class    => [\Admnio\Sunset\Listeners\StoreJob::class],
        \Admnio\Sunset\Events\JobReserved::class  => [\Admnio\Sunset\Listeners\MarkJobAsReserved::class],
        \Admnio\Sunset\Events\JobReleased::class  => [\Admnio\Sunset\Listeners\MarkJobAsReleased::class],
        \Admnio\Sunset\Events\JobCompleted::class => [\Admnio\Sunset\Listeners\MarkJobAsComplete::class],
        \Admnio\Sunset\Events\JobFailed::class    => [\Admnio\Sunset\Listeners\MarkJobAsFailed::class],
    ];
    foreach ($ourMap as $event => $listeners) {
        $events->forget($event); // clears any prior registrations (incl. Horizon's)
        foreach ($listeners as $listener) {
            $events->listen($event, $listener);
        }
    }
});
```

### Horizon listener suppression — concrete approach

Laravel's event dispatcher's `forget($event)` removes ALL listeners registered
for that event. We use the following sequence in `app->booted`:

1. Record the list of Sunset events we own and the Sunset listeners we want on each.
2. For each event:
   - `$events->forget($eventClass)` — clears Horizon's registration plus
     anything else.
   - Re-`listen(...)` our listeners for that event (since the `forget` cleared them too).
3. For the two Laravel stock events we hook (`JobProcessed`, `JobFailed`):
   - We do NOT `forget` these (other Laravel apps may legitimately listen on them).
   - Instead, register our `Translate*` listeners with high priority so they
     run first. Horizon's `MarkJobAsComplete` / `MarkJobAsFailed` listeners
     still receive the events, but they go through Horizon's `JobRepository`
     contract — which is bound to our adapter — which routes to our
     `JobRepository` writing to `sunset:*` keys. End result: idempotent
     recording (the Horizon listener overwrites the same data our listener
     just wrote). Acceptable redundancy in v0.4.0; cleaned up in v0.5.0.

A regression test (`test_horizon_listeners_are_not_registered_for_sunset_events`)
in `SunsetServiceProviderTest` asserts the listener chain for each of the
seven Sunset events contains only `Admnio\Sunset\Listeners\*` classes. This is
the canary that catches breakage if Horizon adds a new listener in a future
release.

### Fallback if `forget()` proves fragile

If a future Horizon version registers listeners using a path that survives
`forget()`, the fallback is:
- Re-register Sunset listeners with high priority that call
  `$event->propagationStopped = true` after dispatching (Laravel 11+
  supports this on the dispatcher level).
- The test above catches the failure mode immediately so we'd notice in CI.

## JobPayload

`Admnio\Sunset\JobPayload` mirrors `Laravel\Horizon\JobPayload`'s public
surface. Same fields and methods:

```php
final class JobPayload
{
    public string $value;
    public array $decoded;

    public function __construct(string $value) { ... }
    public function id(): string;
    public function decoded(): array;
    public function commandName(): string;
    public function tags(): array;
    public function prepare(mixed $job = null): self;        // returns a clone w/ tags + sunset_id injected
    public function nextDelay(): ?int;
}
```

Internally JobPayload stamps the JSON payload with a `sunset_id` key (the
opaque job ID our `JobRepository::nextJobId()` produced). It also normalizes
tags (string-or-array → array of strings) and the `display_name`. Same data
contract as Horizon's — we just stop importing Horizon's class.

The legacy `id` key (Horizon's `Laravel\Horizon\JobPayload::prepare` writes
`id` and `tags` to the payload) stays in the JSON shape so worker code that
reads `$job->payload()['id']` keeps working. We add `sunset_id` alongside,
synced to the same value; downstream code can migrate to that key over the
v0.4 → v0.5 transition.

## Transport changes

### `RedisQueue::pushRaw` (before)

```php
$prepared = (new \Laravel\Horizon\JobPayload($payload))->prepare($this->lastPushed);
$this->raiseHorizonEvent(new \Laravel\Horizon\Events\JobPending($prepared->value), $queue);
$result = parent::pushRaw($prepared->value, $queue, $options);
$this->raiseHorizonEvent(new \Laravel\Horizon\Events\JobPushed($prepared->value), $queue);
return $prepared->id();
```

### `RedisQueue::pushRaw` (after)

```php
$prepared = (new \Admnio\Sunset\JobPayload($payload))->prepare($this->lastPushed);
$connection = $this->getConnectionName();
$queueName  = \Illuminate\Support\Str::replaceFirst('queues:', '', $queue ?? $this->default);

event(new \Admnio\Sunset\Events\JobQueueing($connection, $queueName, $prepared));
$result = parent::pushRaw($prepared->value, $queue, $options);
event(new \Admnio\Sunset\Events\JobQueued($connection, $queueName, $prepared));
return $prepared->id();
```

Same shape of edit in `SqsQueue::pushRaw` (and the receive-side for
`JobReserved`). The transport changes are mechanical — no logic moves, only
event class + payload class swap.

## `sunset:migrate-horizon-keys` command

Signature:
```
sunset:migrate-horizon-keys
    {--dry-run : List keys that would change without writing}
    {--from=horizon : Source prefix (default: horizon)}
    {--to=sunset   : Destination prefix (default: sunset)}
    {--connection= : Redis connection (default: sunset.redis_connection)}
```

Algorithm:
1. Resolve the Redis connection from `--connection` or `sunset.redis_connection`.
2. SCAN keys matching `{from}:*` in chunks of 500.
3. For each key, compute the target name (`{to}:` + suffix).
4. If `--dry-run`: print `RENAME {source} → {target}` and continue.
5. Otherwise: capture TTL with `TTL`, `RENAME` the key, `EXPIRE` to restore TTL
   if it was set. (`RENAME` preserves TTL by default; the explicit dance is
   defensive against future Redis behavior changes.)
6. If the target already exists, log a warning and skip (operator can manually
   resolve).
7. Track and print summary at the end: `Renamed N, skipped M, errored K`.

Idempotent (running twice is a no-op). Reversible (`--from=sunset --to=horizon`
swaps the prefixes).

Special case: keys like `horizon:{job-id}` (per-job hash, no explicit
`horizon:job:` prefix in Horizon's current schema — Horizon stores per-job
hashes at `horizon:{id}`). Our target schema uses `sunset:job:{id}` — the
explicit `:job:` infix prevents collisions with index keys. The migrate
command detects per-job hashes by checking the key's TYPE (`HASH`) and the
presence of `payload` / `display_name` fields, and rewrites them to
`sunset:job:{id}`. All other keys take a straight prefix swap.

## Testing strategy

### Unit tests — pure (no Redis required, Mockery)

- `tests/Unit/Events/JobEventTest.php` — base class carries
  `connectionName`/`queue`/`payload`.
- `tests/Unit/Listeners/<each>Test.php` — verify each listener calls expected
  repo methods with expected args. One test file per listener (9 files).
- `tests/Unit/Adapters/Horizon/<each>Test.php` — delegates each Horizon
  method correctly to our sub-repos; wraps `Laravel\Horizon\JobPayload` →
  `Admnio\Sunset\JobPayload`. One test file per adapter (3 files).
- `tests/Unit/JobPayloadTest.php` — `id()`, `prepare()`, `decoded()`, `tags()`
  behavior; verifies `sunset_id` key is stamped on the JSON value.

### Unit tests — Redis-backed (single class, real Redis)

- `tests/Unit/Repositories/Redis/Redis<each>RepositoryTest.php` — exercise the
  full surface of each repo against a real Redis container. Pattern matches
  v0.3.0's `RedisQueueTest` (under `tests/Unit/` even though Redis is required —
  the class under test is the only Sunset class involved). Four files.

### Integration tests (full Laravel stack + real Redis)

- `tests/Integration/SunsetLifecycleTest.php` — push a job via either
  transport, run the worker, assert pending → reserved → completed records
  appear in `sunset:*` keys.
- `tests/Integration/HorizonDashboardCompatTest.php` — push a job, GET
  `/horizon/api/jobs/pending`, assert the response contains our job (proves
  the adapter chain). Same pattern as v0.3.0's
  `RedisRoundtripTest::test_dashboard_pending_endpoint_lists_redis_job`.
- `tests/Integration/MigrateHorizonKeysTest.php` — seed legacy `horizon:*`
  keys, run `sunset:migrate-horizon-keys`, assert keys moved to `sunset:*`
  with TTLs preserved. Plus a `--dry-run` test.

### Regression assertions

- All 66 v0.3.0 tests must pass with only mechanical edits to two test files
  (`RedisQueueTest`, `SqsQueueTest`) that switch their event assertions from
  `Laravel\Horizon\Events\*` to `Admnio\Sunset\Events\*`.
- `SunsetServiceProviderTest::test_horizon_listeners_are_not_registered_for_sunset_events`
  — resolves the dispatcher, calls `getListeners` for each of the seven
  Sunset events we own, asserts the class list contains only
  `Admnio\Sunset\Listeners\*`, never `Laravel\Horizon\Listeners\*`.

## Open implementation questions (resolve during plan)

- **`MonitorTag` listener:** Horizon's version listens on `JobPushed`, not
  `JobPending`. Listening on `JobQueueing` is cleaner (we can monitor before
  the queue accepts the push) but check the integration test passes — if the
  Horizon dashboard's "monitor this tag" flow depends on a `JobPushed`-timed
  monitor write, switch to `JobQueued`.
- **Snapshot scheduling:** Horizon registers `horizon:snapshot` on a 5-minute
  schedule via its scheduler. We'd register `sunset:snapshot` on the same
  cadence in `boot()` — using the same scheduler hook we use for
  `sunset:sweep-delayed`. Trivial; mentioned here so it doesn't get missed.
- **Failed jobs DB table:** Laravel's `failed_jobs` DB table is unchanged.
  `FailedJobRepository::failed()` writes to both Redis (our index) and the DB
  (via the standard failed-job handler). The DB write is unchanged from
  Laravel's stock path — we don't touch the failer.
