# Sunset v0.8.0 Design Spec — Dashboard SPA + Drop Horizon Dependency

**Date:** 2026-05-19
**Status:** Approved for implementation planning.
**Predecessor:** v0.7.0 (rate limits) — merged 2026-05-19.
**Successor:** None planned at brainstorm time. v1.0.0+ items are deferred follow-ups (see Out of scope).
**Companion direction:** `docs/superpowers/specs/2026-05-19-v1.0.0-dashboard-ui-direction.md` — the pre-brainstorm UI direction constraint. Still applies to this spec (the version label v1.0.0 in that file's name is historical; the constraints inside are referenced as locked design input here).

## Goal

Ship the Sunset SPA dashboard AND drop the `laravel/horizon` composer dependency in a single release. After v0.8.0:

- Sunset has its own self-contained operator UI at `/sunset`, distinct from Horizon's visual identity.
- `composer require admnio/sunset` does NOT pull in `laravel/horizon` (direct or transitive).
- Consumers who install Horizon separately experience no conflict — both `/sunset` and `/horizon` work independently. If a consumer runs both Sunset and Horizon workers, they may compete for jobs; that is the consumer's responsibility, not ours.

This is the final piece of the "Sunset as a standalone Horizon replacement" journey that v0.4.0 (job lifecycle) and v0.5.0 (supervisor) explicitly deferred.

## Decisions locked during brainstorming

| Question | Answer |
|----------|--------|
| Frontend stack | Inertia.js + Vue 3 + Vite + TailwindCSS + Pinia |
| Live updates | Polling (configurable interval, default 3s). No SSE/WebSockets in v0.8 |
| Data layer pattern | Same route returns Inertia render OR JSON for `?refresh=1` polling |
| Auth | `Sunset::auth(fn ($req) => bool)` single gate; default localhost-only in non-local env |
| Visual identity | "Operator Console" — slate base + amber accent, monospace data, terminal-flavored, distinct from Horizon's purple/sidebar look |
| Navigation | Left rail with three grouped sections (Overview, Jobs, Ops); collapses to hamburger on mobile |
| Failed-jobs page pattern | Master–Detail (list left, selected job's details right) |
| Theming | Light + dark via `prefers-color-scheme`; user-overridable via header toggle (persisted to localStorage) |
| Mobile responsiveness | Full responsive down to phone — tables collapse to cards, master-detail collapses to single-pane with back navigation |
| Destructive actions | Retry / delete failed jobs (single + bulk); pause / resume supervisors |
| Horizon dependency | Removed in this same release. Adapters deleted. Native Sunset contracts only |
| Scope decomposition | Originally split between v0.8 (dashboard) and v1.0 (drop Horizon); collapsed back into a single v0.8 release after user confirmed no migration concerns (package is pre-release) |
| Build order | Single feature branch, one PR, ship when complete. No intermediate releases |

## Out of scope (deferred to future releases)

- Worker CPU/memory monitoring (v1.1)
- Realtime worker activity stream / push updates (v1.2)
- Dashboard-side per-queue pause controls (v1.3) — `Sunset::for(...)` rate limits are read-only on the dashboard in v0.8
- Cancel/kill running jobs from the UI (signal safety not worth the complexity)
- Internationalization (English only)
- Full WCAG accessibility audit (basic semantic HTML + keyboard nav only)
- Live rate-limit override or editing from the dashboard
- Dashboard observability hookup to the `sunset:rl:rejects:*` counters introduced in v0.7.0 — the rate-limits page consumes them, but no system-wide telemetry export

## Architecture

```
Consumer Laravel app
  │
  ├─ config/sunset.php             (path, dashboard.poll_interval_seconds, etc.)
  ├─ AppServiceProvider::boot()    Sunset::auth(fn ($req) => …)
  │
  ▼
SunsetServiceProvider
  ├─ Manager singleton             (gains auth() / check() in v0.8)
  ├─ All native repositories       (JobRepository, FailedJobRepository, MetricsRepository,
  │                                  TagRepository, SupervisorRepository,
  │                                  MasterSupervisorRepository, ProcessRepository,
  │                                  SupervisorCommandQueue, LimitRegistry,
  │                                  WorkloadRepository  ← NEW native contract)
  ├─ NO Horizon contract bindings  (adapters under src/Adapters/Horizon/ deleted)
  └─ loadRoutesFrom routes/sunset.php
  │
  ▼
/sunset/* routes (under Authorize middleware, gated by Manager::check())
  │
  ▼
Page controllers under src/Dashboard/Http/Controllers/
  Each controller:
    1. buildProps($request, ...repositories...): array
    2. if $request->query('refresh') === '1' || wantsJson() → JsonResponse(['props' => $props])
       else → Inertia::render('Sunset/<PageName>', $props)
  │
  ▼
Inertia client (resources/js/dashboard/)
  - Vue 3 components per page
  - usePolling() composable refreshes the current page's URL with ?refresh=1
  - Pinia stores for cross-page state (theme, poll coordination, supervisor banner)
```

### Controller contract

Every GET controller follows this pattern:

```php
final class FailedJobsController extends Controller
{
    public function show(
        Request $request,
        FailedJobRepository $failed,
        TagRepository $tags,
    ): Response|JsonResponse {
        $props = [
            'failures' => $failed->paginate(...),
            'tags'     => $tags->knownTags(),
            'selected' => $this->resolveSelected($request, $failed),
            // ...
        ];

        if ($request->query('refresh') === '1' || $request->wantsJson()) {
            return response()->json(['props' => $props]);
        }
        return Inertia::render('Sunset/Failed', $props);
    }
}
```

The `props` array shape is identical on initial Inertia render and on `?refresh=1` JSON polling. This is contractually load-bearing — the SPA uses the same Vue component for both.

### Authorize middleware

```php
final class Authorize
{
    public function __construct(private Manager $manager) {}

    public function handle(Request $request, Closure $next): mixed
    {
        return $this->manager->check($request) ? $next($request) : abort(403);
    }
}
```

`Manager::check($request)` was added in this release. Returns:
- `true` if a registered `Sunset::auth()` callback returns truthy
- `true` if `app.env === 'local'`
- `true` for `127.0.0.1` / `::1` regardless of env (safety net for first-run)
- `false` otherwise

### Polling

`config/sunset.php` gains:
```php
'dashboard' => [
    'path'                  => env('SUNSET_PATH', 'sunset'),
    'poll_interval_seconds' => (int) env('SUNSET_DASHBOARD_POLL', 3),
],
```

Frontend reads `poll_interval_seconds` from a server-rendered config blob in the Inertia root view. `usePolling()` debounces to a single in-flight request per page.

## Frontend

### Stack

- **Inertia.js 1.x + Vue 3** — server-driven page routing, client-rendered components.
- **Vite 5** — single bundle output to `public/vendor/sunset/app.{js,css}`. Source maps NOT distributed.
- **Tailwind CSS 3** — utility-first; custom CSS variables drive light/dark theme switching.
- **Pinia 2** — cross-page state (theme, poll coordination, command-palette).
- **fuse.js** (or similar small fuzzy-matcher) — command palette search.

### Visual system

| Aspect | Token |
|--------|-------|
| Font (data) | `ui-monospace, "Cascadia Code", Menlo, monospace` at 13px |
| Font (chrome) | `system-ui, -apple-system, sans-serif` at 12-14px |
| Dark base / rail / card | `#11161d` / `#0d1218` / `#0a0d12` |
| Light base / rail / card | `#fafaf7` / `#f5f4ef` / `#ffffff` (warm-tinted whites) |
| Primary accent | `#f59e0b` (amber) — active nav, focus rings, primary actions |
| Status: error/failed | `#ef4444` |
| Status: success/admitted | `#22c55e` |
| Status: warning/throttled | `#f59e0b` |
| Status: info/pending | `#60a5fa` |
| Border (dark / light) | `#1f2630` / `#e7e5dc` |
| Density | 28px table row, 13px data, 11px chrome |

All colors flow through CSS custom properties (`--sunset-bg`, `--sunset-accent`, etc.) so theme switching is a single class flip on `<html>`.

### Theming

`useTheme()` composable:
1. Reads `localStorage['sunset.theme']` for an override (`'light'` | `'dark'` | unset)
2. Falls back to `window.matchMedia('(prefers-color-scheme: dark)')`
3. Listens for OS theme changes when no manual override is set
4. Header toggle sets the override and immediately applies the class

### Responsive breakpoints

| Width | Behavior |
|-------|----------|
| ≥ 1024px | Full left rail (160px wide); tables in full |
| 640-1024px | Left rail collapses to icon-only (48px); tables horizontal-scroll |
| < 640px | Left rail becomes a hamburger drawer; tables collapse to stacked cards (each row = a card with label-value pairs); master-detail becomes single-pane with back navigation; touch targets ≥ 44px; primary buttons full-width |

### Command palette (⌘K / Ctrl+K)

Fuzzy search across:
- All 12 pages (by name and aliases)
- All known queues (from `WorkloadRepository`)
- Recent failed-job IDs / tags
- Active supervisors

Keyboard-first: ↑/↓ or j/k navigates, Enter opens, Esc closes. Exposed in the header on every page.

### Failed-jobs Master-Detail

Three-column layout at ≥ 1024px:
- Left rail (160px nav)
- Middle list (200-260px, scrollable, compact rows)
- Right detail panel (flex-grow, scrollable, full exception + actions)

On selection, the right panel updates without a page reload (Inertia partial reload of just the `selected` prop). Retry/delete actions appear in the panel as buttons; bulk actions live in a toolbar above the list.

### Asset distribution

`sunset:install` artisan command (extended from its v0.5.0 form):
1. Publishes `config/sunset.php` (with `--force` confirmation if a config exists)
2. Publishes the compiled JS+CSS bundle to `public/vendor/sunset/`
3. Optionally publishes the Inertia root view `resources/views/sunset-app.blade.php` to `resources/views/vendor/sunset/` for consumer override
4. Prints a "Visit /sunset" success message

### New contract added

```
src/Contracts/WorkloadRepository.php   ← NEW (interface)
```

Mirrors the shape Sunset's existing `SunsetWorkloadRepository` exposes. Sunset now owns this contract; consumers depend on `Admnio\Sunset\Contracts\WorkloadRepository`, not Horizon's.

### Tests removed

Any test that mounts `Laravel\Horizon\*` routes or asserts adapter behavior:
- `tests/Unit/Adapters/Horizon/*` (if any exist)
- Any integration test referencing `/horizon/*` endpoints
- Any test asserting the Horizon contract bindings exist in the container

### Composer behavior

After v0.8.0:
- `composer require admnio/sunset` installs Sunset alone. No `laravel/horizon` pulled.
- A consumer who runs `composer require laravel/horizon` separately gets it as an independent peer. Both Horizon's `HorizonServiceProvider` and Sunset's `SunsetServiceProvider` register their own bindings and routes without overlap.
- Running both `php artisan horizon` and `php artisan sunset:work` simultaneously will cause both supervisors to compete for the same queues if pointed at the same connection — documented as out of scope; user's responsibility.

## Testing strategy

| Layer | Coverage |
|-------|----------|
| Unit | Each controller's `buildProps()` against in-memory repositories returns the expected shape |
| Unit | `Authorize` middleware: local env → allow; production + localhost IP → allow; production + remote IP → deny; custom `Sunset::auth()` callback → wins |
| Unit | `Manager::auth()` callback registration; `Manager::check()` default behavior |
| Unit | Polling-response shape stability — `?refresh=1` JSON has identical keys to initial Inertia props |
| Unit | New `WorkloadRepository` contract + `SunsetWorkloadRepository` implementation tests still pass |
| Unit | New native `JobPayload` (prepare/id/value/decoded) — behavior matches the v0.5.0-era extends-Horizon behavior |
| Unit | New native `AutoScaler` — scaling math equivalent to the v0.5.0 implementation |
| Unit | `tests/Unit/Dashboard/Routes/RouteResolutionTest.php` — every route in `routes/sunset.php` resolves |
| Integration | Real Redis: push 3 failed jobs via `FailedJobRepository` → GET `/sunset/jobs/failed?refresh=1` → JSON contains all three with the right shape |
| Integration | POST `/sunset/jobs/failed/{id}/retry` → re-dispatches; POST `/sunset/jobs/failed/delete` (bulk) → removes from the store and from Laravel's `failed_jobs` table |
| Integration | POST `/sunset/supervisors/{name}/pause` → command lands in `SupervisorCommandQueue`; subsequent `/sunset/supervisors` reflects the paused state |
| Integration | Composer dry-run: `composer install` in a fresh Laravel app requiring `admnio/sunset` does NOT install `laravel/horizon` |
| Integration | Consumer-installs-Horizon-themselves smoke test: install both packages, confirm `/sunset` and `/horizon` both work, no binding conflicts |
| Browser (Dusk) | One smoke test per page: page renders, left rail visible, expected title present |
| Browser (Dusk) | Failed-jobs master-detail interaction: click row → details update → click retry → toast confirms → row removed |
| Browser (Dusk) | Theme toggle: click → CSS variable flips → reload preserves choice |
| Browser (Dusk) | Mobile breakpoint at 375px: hamburger nav, stacked cards, master-detail single-pane with back button |
| Browser (Dusk) | Command palette: ⌘K opens; type "fail" → "Failed Jobs" matches; Enter navigates |

## Error handling

| Failure | Behavior |
|---------|----------|
| Redis unreachable on polling fetch | "Unable to refresh" toast; previous data stays; auto-retry on next cycle |
| Auth gate throws | Deny (403); exception logged |
| Destructive action fails server-side | Error toast; affected row not optimistically removed |
| Retry target doesn't exist (already gone) | 404 JSON; toast "Job no longer available"; row removed |
| Supervisor command-queue write fails | Error toast; supervisors page reflects actual state on next poll |
| Consumer installs Horizon separately | No conflict — Sunset registers no Horizon bindings |
| `app.key` missing in consumer app | `sunset:install` surfaces a clear error pre-publish |
