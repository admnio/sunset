# Sunset v1.3.0 — Queue Pause / Resume Controls

## Goal

Let operators pause individual queues from the dashboard without pausing entire supervisor process trees. Useful for: throttling a misbehaving job class without affecting unrelated work; coordinating a downstream maintenance window; emergency stop on a queue that's pulling poisoned payloads. Works uniformly across SQS, Redis, and RabbitMQ transports.

## What "pause a queue" means

Operationally, pausing the queue `(connection, queue)`:
1. Workers will not pop new jobs from that queue. They sleep their normal idle cadence and check again.
2. In-flight jobs (already popped, currently executing) continue to completion. Pause is "stop popping," not "kill running."
3. Producers can still enqueue. New jobs accumulate on the queue. The queue is paused, not closed.
4. Once resumed, workers resume popping on the next loop tick.

Pause is a soft signal. The check happens at the transport's `pop()` boundary. There's an inherent race: a worker that popped a job a microsecond before the pause flag was set will still process that job. Operators should expect "pause takes effect within one pop cycle" (≤ worker sleep interval, default 3s).

## Non-goals

- Pausing producers. If your app pushes jobs to a paused queue, they accumulate. We don't add producer-side guards; that's a separate "circuit breaker" feature.
- Pausing a supervisor (already covered by the existing pause/resume buttons since v0.5.0). Supervisor pause stops ALL workers in that supervisor; queue pause is finer-grained.
- Cron / time-window scheduling of pauses ("auto-pause from 22:00 to 06:00"). Out of scope for v1.3.0 — operators can wire a Laravel scheduled command if needed.
- Per-job-class pausing. Use rate-limits with `->drop()` for that.

## Architecture

**Storage.** A single Redis SET `sunset:queues:paused` holds `"{connection}:{queue}"` strings. Operations:
- `SADD sunset:queues:paused "{conn}:{queue}"` — pause
- `SREM sunset:queues:paused "{conn}:{queue}"` — resume
- `SISMEMBER sunset:queues:paused "{conn}:{queue}"` — is paused?
- `SMEMBERS sunset:queues:paused` — full list (dashboard render)

No TTL — pauses persist until explicitly resumed. The set is uncapped (operators won't have thousands of paused queues; if they do, the SISMEMBER cost is still O(1)).

**Gate.** New `QueuePauseGate` (`@internal`) wraps the repository. Single method `isPaused(string $connection, string $queue): bool`. Failure mode: when Redis is unreachable, gate **fails open** (returns false) so a Redis outage doesn't silently stop the entire fleet. Logs at warning level on each failure (rate-limited to once per 60s via in-process throttle to avoid log spam).

**Transport integration.** Each transport's `pop()` method gets one new line at the top, mirroring the existing rate-limit gate pattern:

```php
public function pop($queue = null)
{
    if ($this->container->make(QueuePauseGate::class)
            ->isPaused($this->getConnectionName(), $queue ?: $this->default)) {
        return null;
    }
    // ...existing pop logic
}
```

Three transports get this: `SqsQueue::pop()`, `RedisQueue::pop()`, `RabbitQueue::pop()`. Same shape in each.

Why this layer (not a Looping event listener)? The Looping event fires BEFORE the worker decides which queue to pop from. By the time we know the queue name, we're inside the transport's pop(). The transport is the natural integration point and it's already where the rate-limit gate sits.

**Public API additions** (committed for v1.x):
- `Admnio\Sunset\Contracts\QueuePauseRepository` — read+write contract:
  - `pause(string $connection, string $queue): void`
  - `resume(string $connection, string $queue): void`
  - `isPaused(string $connection, string $queue): bool`
  - `all(): array` — `list<array{connection: string, queue: string}>`
- `Admnio\Sunset\Events\QueuePaused` — public event, fired AFTER successful pause. Properties: `connection`, `queue`.
- `Admnio\Sunset\Events\QueueResumed` — public event, fired AFTER successful resume. Properties: `connection`, `queue`.

The repository contract has both read AND write methods (unlike `WorkerMetricsRepository` and `ActivityRepository` which separated write into an internal method). Reason: pause/resume are explicit user-driven actions, not implicit recording. Consumers might want to script a pause via the contract — making write public is the right call.

**Dashboard.** The existing `/sunset/workload` page already lists queues with pending counts. Extend it:
- Add a paused indicator column. When `paused === true`, render a yellow "PAUSED" pill instead of the normal pending-count.
- Add per-row pause/resume button next to each queue. Button label flips based on current state.

New POST routes (under the existing dashboard prefix + Authorize middleware):
- `POST /sunset/workload/{connection}/{queue}/pause` — pauses, fires `QueuePaused`, redirects back.
- `POST /sunset/workload/{connection}/{queue}/resume` — resumes, fires `QueueResumed`, redirects back.

Route parameters need URL-safe encoding for queue names with slashes/colons (SQS allows `-`, `_`, `.`, but not `/` or `:` typically). Use Laravel's URL-encoding on the dashboard side; the routes accept `{connection}` and `{queue}` as path segments with `where` constraints that allow any non-empty string except `/`.

**Artisan commands** (`@internal` classes, public command names):
- `sunset:pause-queue {connection} {queue}` — pause from CLI
- `sunset:resume-queue {connection} {queue}` — resume from CLI

Useful for ops scripting outside the dashboard.

**Activity stream integration.** When `QueuePaused`/`QueueResumed` fire, the recorder picks them up if we extend the `ActivityEventFactory` to translate them. **Yes, do this** — pause/resume is high-signal activity operators want to see in the live log. Add two new event types: `queue_paused` and `queue_resumed`. Payload: `{connection, queue, actor: 'dashboard'|'cli'|null}`.

The `actor` field is a string the caller passes when firing the event. The dashboard controller passes `'dashboard'`; the artisan commands pass `'cli'`; programmatic API consumers pass `null`. Lets operators trace where a pause came from when reviewing the activity log.

## Cross-platform notes

Pure Redis SET operations. Works identically on Linux, macOS, Windows, Octane, FPM, RoadRunner. No platform-specific concerns.

## Failure semantics

- **Redis down during pop()**: gate fails open (returns false) → workers continue popping → jobs continue processing. Pause information is unavailable but the queue stays functional. We pick fail-open over fail-closed because the alternative is fleet-wide stoppage on a Redis blip, which is worse than the small window where a recently-paused queue keeps popping.
- **Redis down during pause()/resume()**: throws. The caller (dashboard or artisan) sees the error and can retry. No silent failure.
- **Concurrent pause + worker pop**: race exists but harmless. The worst case is one extra job pops after the pause is recorded. Documented as expected.
- **Resume of an unpaused queue**: SREM on a non-member is a no-op in Redis. `QueueResumed` event still fires. Idempotent.
- **Pause of an already-paused queue**: SADD on existing member is a no-op. `QueuePaused` event fires again. Idempotent. Activity log gets a duplicate entry; that's fine — accurately reflects the operator's action.

## Testing strategy

- **Unit:** Gate fails open on Redis throw + log warning. Factory translates `QueuePaused` / `QueueResumed`. Events have the right properties.
- **Integration:** Repository against real Redis — pause/resume/isPaused/all in all combinations including idempotence + cross-connection isolation. Each transport's pop() returns null when the queue is paused (use a fake transport client to assert pop() doesn't reach the underlying SQS/Redis/Rabbit client). Controller POSTs pause/resume and asserts the repo state changed + the event fired. Workload page shows paused indicator.
- **No browser test.** Hand-wave the smoke test in maintainer release notes.
