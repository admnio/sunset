# Sunset v1.1.0 — Worker CPU/Memory Monitoring

## Goal

Surface per-worker CPU and memory usage in the Sunset dashboard so operators can see when a worker is leaking, runaway, or under-utilised — without bolting on a separate APM.

## Non-goals

- Per-job CPU/memory profiling. The job lifecycle already records duration; per-job memory deltas are noisy enough to mislead and the cost (sampling getrusage twice per job) outweighs the value.
- Aggregate "total worker fleet" charts. That's a follow-up; v1.1.0 ships per-worker only.
- Alerting / thresholds. Out of scope — the dashboard surfaces data, users wire it to whatever paging system they already have.

## Architecture

**Sampling.** Hook Laravel's `Illuminate\Queue\Events\Looping` event (fires once per worker loop tick, just before pop). The listener reads `memory_get_usage(true)` and `getrusage()`. It throttles to once per `sunset.telemetry.interval_seconds` (default 5) using an in-process `static float $lastReportAt`. The loop already fires every few hundred ms even when idle (Laravel's `sleep` between polls), so 5s throttling means ~12 samples/minute per worker. Skips entirely when `sunset.telemetry.enabled` is false.

**CPU calculation.** Sample N: `(ru_utime_N + ru_stime_N)`. CPU% over the window = `(cpu_N - cpu_{N-1}) / (wall_N - wall_{N-1}) * 100`. First sample reports cpu_pct=null. Cross-platform: `getrusage()` returns zeros for ru_utime/ru_stime on Windows — listener writes `cpu_pct: null` when delta == 0 and previous sample also was 0, and Health page already shows a Windows-compat banner.

**Storage layout** (Redis, under existing `sunset:` prefix):

- `sunset:worker_metrics:{pid}` — hash with current snapshot (`pid, supervisor, connection, queues, started_at, rss_bytes, cpu_pct, jobs_processed, last_report_at`). TTL 30s so dead workers fall off without a sweep.
- `sunset:worker_metrics:{pid}:series:rss` — sorted set, score=unix timestamp, member=rss bytes. Capped to 60 entries via `ZREMRANGEBYRANK ... 0 -61` per write.
- `sunset:worker_metrics:{pid}:series:cpu` — sorted set, score=unix timestamp, member=cpu_pct * 100 (int, to keep series members unique-ish).
- `sunset:worker_metrics:pids` — set of currently-known PIDs (for SCAN-free read on the dashboard side).

The 30s hash TTL is the source of truth for "live"; the sweep command (Task 6) reconciles dead-but-not-yet-expired entries every minute to keep the PIDs set clean.

**Public API additions.** All other classes stay `@internal`.

- `Admnio\Sunset\Contracts\WorkerMetricsRepository` (read API). Methods: `all(): array`, `find(int $pid): ?WorkerMetricsSnapshot`, `series(int $pid, string $kind, int $maxPoints = 60): array`.
- `Admnio\Sunset\Telemetry\WorkerMetricsSnapshot` (DTO — value object, public). Properties: `pid, supervisor, connection, queues, startedAt, rssBytes, cpuPct, jobsProcessed, lastReportAt`.
- New config keys under `sunset.telemetry.{enabled, interval_seconds, series_points}`.

**Dashboard surface.** Enhance the existing `/sunset/supervisors` Inertia page (`SupervisorsController`) to include per-process metric snapshots and short sparkline data. No new route. The frontend renders two new columns (RSS, CPU%) and a per-row expandable sparkline. Polling already in place via the shared `usePolling()` composable picks up updates.

**Failure semantics.** If Redis is down, the listener silently swallows the exception (telemetry is observability, not load-bearing). The repository's read path returns empty arrays on Redis failure — Supervisors page degrades to "—" cells.

## Cross-platform notes

- **Linux/macOS:** Full `getrusage()` support — CPU% reflects user+sys time deltas.
- **Windows:** `getrusage()` returns ru_utime=0/ru_stime=0 on Windows ([PHP docs](https://www.php.net/manual/en/function.getrusage.php)). Listener detects this (`cpu_delta == 0` for two consecutive samples) and writes `cpu_pct: null`. Memory always works.
- **Octane:** Same listener works under Octane — the static `$lastReportAt` is per-worker-process and gets reset between requests doesn't matter because Octane processes don't run `php artisan sunset:worker`; they're separate HTTP workers. The Octane app process never hits the Looping event because Octane doesn't dispatch queues in-process.

## Testing strategy

- **Unit:** Sampler with a fake clock proves throttling + CPU delta math. Listener tested with a mock Sampler + mock repository.
- **Integration:** Real Redis (already in docker-compose); end-to-end `Looping` event → sampler → Redis → read repository → controller payload.
- **No browser test.** The visual rendering of two new columns + a sparkline is covered by the existing Supervisors-page Dusk smoke test plus the props-shape regression test.

## Public-API stability

`WorkerMetricsRepository` contract and `WorkerMetricsSnapshot` value object are committed as stable for v1.x. The `sunset:worker_metrics:*` Redis key shape is internal — consumers depend on the contract, not the raw Redis layout.
