# Sunset v0.2.0 — Rename and Transport Foundation

**Date:** 2026-05-17
**Status:** Draft (awaiting user review)
**Package:** `admnio/sunset` (renaming from `masonworkforce/horizon-sqs`)
**Sub-project of:** Sunset for Laravel — a Horizon replacement

## Roadmap Context

Sunset is a planned multi-release Horizon replacement. This spec covers **only v0.2.0**. The broader roadmap, agreed during brainstorming:

| Version | Scope |
|---|---|
| **v0.2.0 (this spec)** | Rename `horizon-sqs` → Sunset + `Transport` interface + refactor SQS code behind it |
| v0.3.0 | Sunset's own Redis transport |
| v0.4.0 | `sunset:work` supervisor replacing `php artisan horizon` |
| v0.5.0 | Sunset repositories replacing Horizon's JobRepository/MetricsRepository/etc. |
| v1.0.0 | Full SPA dashboard + drop `laravel/horizon` from composer.json |
| v1.1.0 | Worker CPU/Memory monitoring |
| v1.2.0 | Realtime worker activity stream |
| v1.3.0 | Queue pause/resume controls |

Each row is its own spec → plan → implementation cycle.

## Goal

Rename `masonworkforce/horizon-sqs` to `admnio/sunset` and introduce a `Transport` abstraction so future transports (Redis, BullMQ, RabbitMQ, LavinMQ) can plug in without touching SQS-specific code.

## Acceptance Criteria

1. Package installs as `admnio/sunset` via Composer.
2. All classes live under the `Admnio\Sunset\` namespace.
3. A `Admnio\Sunset\Contracts\Transport` interface exists and is implemented by `Admnio\Sunset\Transports\Sqs\SqsTransport`.
4. The full pre-existing test suite (37 unit + 9 integration = 46 tests) passes after path/namespace updates, with no behavior changes.
5. Two new unit test files cover `SqsTransport` and `TransportRegistry`.
6. A `UPGRADING.md` at project root documents the migration from `masonworkforce/horizon-sqs` v0.1.x.
7. README is updated to reflect the new name, namespace, env vars, and config key path.
8. `laravel/horizon ^5.20` remains a hard dependency — UI is unchanged.

## Non-Goals (Deferred)

- Building a Redis transport (v0.3.0)
- Replacing `php artisan horizon` supervisor (v0.4.0)
- Replacing Horizon's repositories (v0.5.0)
- Custom dashboard SPA (v1.0.0)
- Worker resource monitoring, real-time visibility, pause/resume (v1.1+)
- Capability discovery methods on the `Transport` interface (`supportsFifo()`, etc.) — added when v1.0.0 dashboard demands them
- Backward-compatibility shims for v0.1.x class names — pre-1.0 BC break is acceptable; migration is documented in `UPGRADING.md`

## Design Decisions

| Decision | Choice | Rationale |
|---|---|---|
| Composer name | `admnio/sunset` | User-specified vendor |
| PHP namespace | `Admnio\Sunset\` | Matches Composer convention; vendor-prefixed reduces collision risk |
| BC compatibility | Hard break, document migration | Pre-1.0 release; aliases would be permanent maintenance burden |
| Horizon dep in v0.2.0 | Keep | UI replacement comes in v1.0.0; transitional dependency is fine |
| Transport interface shape | Thin factory + workload (3 methods) | YAGNI — only consumers in v0.2.0 are the connector and workload repository |
| Where SQS-specific code lives | Under `src/Transports/Sqs/` | Isolation; future `Transports/Redis/` will not cross-import |
| Workload responsibility split | `Transport::workload()` returns data; `SunsetWorkloadRepository` adapts to Horizon's `WorkloadRepository` contract and adds wait-time math + caching | Keeps Horizon-specific knowledge in the adapter, not the transport |

## Transport Interface

```php
<?php

namespace Admnio\Sunset\Contracts;

use Illuminate\Contracts\Queue\Queue;

interface Transport
{
    /** Stable identifier, e.g. 'sqs', 'redis'. Used as the key under config('sunset.transports'). */
    public function name(): string;

    /** Build a Laravel Queue for a given connection config (from config/queue.php). */
    public function connect(array $config): Queue;

    /**
     * Fetch per-queue depth metadata for the dashboard's Workload page.
     *
     * @param list<string> $queues
     * @return list<array{name: string, length: int, processes: int, wait: int, split_queues: null|array}>
     */
    public function workload(array $queues): array;
}
```

Three responsibilities, no more:
- `name()` identifies the transport. Also used as the config key.
- `connect($config)` is what Laravel's `QueueManager` ultimately calls via Sunset's connector. Returns a configured `Illuminate\Contracts\Queue\Queue`.
- `workload($queues)` returns data shaped to match Horizon's `WorkloadRepository` contract output. Wait-time math (length × runtime / processes) is NOT done here — it stays in `SunsetWorkloadRepository`, which is transport-agnostic.

### Component diagram

```
Laravel Queue Manager
    │
    └─► resolves 'sqs' driver via Sunset's connector
            │
            └─► calls Transport::connect($config)
                    │
                    └─► returns Admnio\Sunset\Transports\Sqs\SqsQueue
                                 (the Laravel Queue that runs push/pop)

Horizon Dashboard
    │
    └─► calls WorkloadRepository::get()
            │
            └─► binds to SunsetWorkloadRepository (adapter)
                    │
                    └─► calls Transport::workload($queues)
                            │
                            └─► returns workload records
```

### Transport registry

A small `Admnio\Sunset\Support\TransportRegistry` holds Transports keyed by `name()`. `SunsetServiceProvider::register()` populates it once with the SQS transport. The connector and workload repository resolve the registry and pull the right Transport. v0.3.0 only adds `$registry->register(new RedisTransport(...))` — no other code changes.

### Relationship between `Transport` and Laravel's `ConnectorInterface`

These are two different interfaces serving different layers:

- `Illuminate\Queue\Connectors\ConnectorInterface` is **Laravel's** contract. The `QueueManager` resolves it via `Queue::extend('sqs', fn () => new SqsConnector(...))` and calls `$connector->connect($config)` to get a queue. Sunset must implement this to integrate with Laravel.
- `Admnio\Sunset\Contracts\Transport` is **Sunset's** contract. It owns both the queue construction logic AND the workload-fetching logic for a given transport type. The dashboard reads through it.

In v0.2.0 the SQS `SqsConnector` is a thin shim — its `connect($config)` method delegates to the registered `SqsTransport::connect($config)`. This keeps Laravel's plumbing intact while letting all transport-specific knowledge live in one place (the `Transport` implementation). Future transports add one Transport class; the connector wiring pattern is identical.

```
  Queue/Payload/PayloadEnricher.php         →           Payload/PayloadEnricher.php
  Queue/Payload/ExtendedPayloadHandler.php  →           Payload/ExtendedPayloadHandler.php
  Queue/Delay/DelayedJobStore.php           →           Delay/DelayedJobStore.php

  Repositories/SqsWorkloadRepository.php    →       Repositories/SunsetWorkloadRepository.php
                                                      (now reads via Transport, not SQS directly)
```

Three structural changes worth calling out:

1. **`Transports/Sqs/` is everything SQS-specific.** Payload enricher, S3 spillover, delay sweep, FIFO attributes, the `SqsQueue`, the `SqsConnector` — all under one subtree. When Redis transport lands in v0.3.0, it goes in `Transports/Redis/` and never imports anything from `Transports/Sqs/`.

2. **`SqsTransport` is new.** Small factory class implementing the `Transport` contract: `name()` returns `'sqs'`, `connect()` builds the `SqsQueue` (using current logic from `SqsConnector`), `workload()` does the GetQueueAttributes batch.

3. **`SunsetWorkloadRepository` replaces `SqsWorkloadRepository`.** It resolves the configured `Transport` from the `TransportRegistry` and calls `->workload($queues)`. Wait-time math, cache wrapper, logger behavior, supervisor-process-count lookup — all stay in this class. Only the GetQueueAttributes-specific code moves into `SqsTransport`.

## Migration Guide (UPGRADING.md content outline)

### 1. Composer swap

```bash
composer remove masonworkforce/horizon-sqs
composer require admnio/sunset:^0.2
php artisan vendor:publish --tag=sunset-config
```

### 2. Config rename

```bash
mv config/horizon-sqs.php config/sunset.php
```

Re-apply any local edits to the new file. The shape changed — top-level keys `sqs_max_delay`, `long_delay_sweep_interval`, `visibility_heartbeat`, `fifo`, `extended_payload` all moved under `transports.sqs.*`. Document with a before/after example.

Env vars:
- `HORIZON_SQS_REDIS` → `SUNSET_REDIS`
- `HORIZON_SQS_S3_BUCKET` → `SUNSET_S3_BUCKET`

### 3. Code references in user's app (rare — most users only touch config)

| v0.1.x reference                                          | v0.2.0 replacement                                       |
|-----------------------------------------------------------|----------------------------------------------------------|
| `use MasonWorkforce\HorizonSqs\…`                         | `use Admnio\Sunset\…`                                    |
| `MasonWorkforce\HorizonSqs\HorizonSqsServiceProvider`     | `Admnio\Sunset\SunsetServiceProvider`                    |
| `MasonWorkforce\HorizonSqs\Exceptions\…`                  | `Admnio\Sunset\Exceptions\…`                             |
| custom subclasses of `HorizonSqsQueue`                    | extend `Admnio\Sunset\Transports\Sqs\SqsQueue`           |
| `php artisan horizon-sqs:sweep-delayed`                   | `php artisan sunset:sweep-delayed`                       |

### 4. Data carryover

- **SQS queues:** unchanged. Same queue URLs, jobs in-flight at upgrade-time continue to be picked up.
- **S3 spillover prefix:** default changes (`horizon-sqs-payloads/` → `sunset-payloads/`). Existing buckets keep their old prefix configured — users can either keep `prefix: 'horizon-sqs-payloads/'` in their `config/sunset.php` indefinitely, or do a one-time S3 copy. Document both.
- **Redis sidecar keys:** the long-delay sorted set renames (`horizon-sqs:delayed` → `sunset:delayed`). Ship a `sunset:migrate-redis-keys` artisan command that:
  - Uses Redis `RENAMENX` (rename-if-not-exists) to avoid clobbering an already-populated new key.
  - No-ops when the old key does not exist.
  - Reports old-key count, new-key count, and rename result to stdout.
  - Is safe to run multiple times.
- The cache key `horizon-sqs:workload` is ephemeral (5s TTL); no migration needed.

### 5. Scheduler entries

The package auto-registers the sweep on `Schedule`. Users who copied the registration into their own `Console/Kernel.php` need to change `horizon-sqs:sweep-delayed` → `sunset:sweep-delayed`.

### 6. Horizon config

`config/horizon.php` supervisor blocks with `'connection' => 'sqs'` are unchanged. Sunset still extends Laravel's `sqs` driver.

## Testing Strategy

The refactor lands as a series of small commits, each one keeping `vendor/bin/phpunit` green. Sequence:

1. Rename composer name + namespace in autoload (composer regen) — no logic changes.
2. Move/rename files + update `namespace` / `use` lines (one mechanical pass).
3. Introduce `Transport` interface + `TransportRegistry` + `SqsTransport` (new files).
4. Refactor `SunsetWorkloadRepository` to read via `Transport::workload()`.
5. Update tests to match new paths; add `SqsTransportTest` and `TransportRegistryTest`.
6. Update `UPGRADING.md`, README, config publishing.

### New test coverage in v0.2.0

- `SqsTransportTest`: `name()` returns `'sqs'`; `connect($cfg)` returns an `SqsQueue` configured per the array; `workload(['orders','default'])` issues async GetQueueAttributes and returns shape-correct records (mocked SQS client).
- `TransportRegistryTest`: registering a transport, looking up by name, throwing `InvalidConfigurationException` for unknown names.
- `SunsetWorkloadRepositoryTest`: asserts against a `FakeTransport` (in-memory test double implementing `Transport`) rather than mocking the SQS client directly. Wait-time math + cache + logger behavior remain covered.

### Test counts

- Existing 37 unit + 9 integration = 46 tests stay (paths updated, logic unchanged).
- +2 unit tests for `SqsTransport`.
- +2 unit tests for `TransportRegistry`.
- New total ≈ 50 tests. No new integration tests — the refactor doesn't change external behavior.

### CI matrix

Stays the same: PHP 8.2/8.3/8.4 × Laravel 10/11/12, plus the LocalStack-backed integration job. Update workflow file `composer require` line to use `admnio/sunset` package context if needed.

## Risks and Mitigations

| Risk | Mitigation |
|---|---|
| Mass namespace edit introduces stray `MasonWorkforce\HorizonSqs` references | After the rename pass, run `grep -r "HorizonSqs\|horizon-sqs" src/ tests/ config/` and verify only intentional migration-doc references remain |
| Composer cache holds stale autoload data | Plan includes explicit `composer dump-autoload -o` after rename |
| Users' Horizon configs reference the old class name in custom service-provider order | Documented in UPGRADING.md; rare in practice |
| Redis sorted-set key rename strands buffered delayed jobs | `sunset:migrate-redis-keys` command renames atomically with Redis `RENAME` (or `RENAMENX` to be safe) |
