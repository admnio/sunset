# Sunset v1.0.0 Dashboard — Pre-Brainstorm UI Direction

**Date:** 2026-05-19
**Status:** Design constraint captured in advance of v1.0.0 brainstorming. Not a full spec.
**Predecessor:** v0.5.0 (supervisor subsystem) — both v0.4.0 and v0.5.0 explicitly defer the Sunset SPA dashboard and the `laravel/horizon` composer-dep drop to v1.0.0.

## Why this file exists

When v1.0.0 brainstorming begins, we want this direction locked in before any layout, component, or color choices are debated. It is the kind of constraint that's easy to slide past in a brainstorming session unless it's recorded explicitly.

## Direction

**The Sunset dashboard UI must NOT look like Laravel Horizon's UI.** Sunset is its own product, with its own identity, and the dashboard is the most visible expression of that. A consumer who has used Horizon before should not, at first glance, mistake Sunset for a re-skinned Horizon — and the engineering team should not be tempted to copy-paste Horizon's interface conventions out of muscle memory.

Concretely this means:

- **Distinct visual language.** Don't reuse Horizon's color palette, header bar, sidebar layout pattern, or chart styling. The dashboard should be visually recognisable as Sunset within one second of loading.
- **Distinct information architecture.** Even where pages cover the same conceptual ground (e.g. failed jobs, workload), the layout, hierarchy, and primary actions should be designed for clarity and efficiency first, not for parity with Horizon. Where Horizon's choices are good, we are free to adopt them — but only when they're the best choice on the merits, not by default.
- **Super clean, efficient, intuitive.** The UI is the primary differentiator from Horizon's continuing existence in the Laravel ecosystem. Prioritise low-noise pages, fast scanability, and minimal clicks to take the most common operator actions (retry a failed job, pause a supervisor, inspect a tag).
- **No brand confusion.** Logo, typography, microcopy, and tone should clearly identify Sunset. The dashboard is a marketing surface for the package as much as it is an operator tool.

## Out of scope for this note

- Specific colors, typography, layout grids, or component library choices — those belong in v1.0.0 brainstorming.
- Whether to ship a design system, Tailwind config, or just bespoke CSS — also v1.0.0.
- Frontend framework / build tooling — also v1.0.0.

## How to use this note

When v1.0.0 brainstorming starts:
1. Read this file first.
2. Treat the four bullets above as locked design constraints.
3. The rest of the design (stack, layout, palette, components) is open.
