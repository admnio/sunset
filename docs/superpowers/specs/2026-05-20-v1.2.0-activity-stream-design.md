# Sunset v1.2.0 ‚Äî Realtime Worker Activity Stream

## Goal

Replace 3-second polling with a live event ticker on a new Activity page so operators see job failures, worker restarts, and rate-limit rejections the instant they happen, not on the next poll tick.

## Non-goals

- Realtime updates to every existing page (Failed Jobs list, Workload, etc.). Those keep their existing polling. Only the Activity page is live; the other pages still poll, by design ‚Äî they show counts and aggregates that don't benefit from sub-second freshness.
- WebSockets / Laravel Reverb. SSE is enough ‚Äî one-way server‚Üíclient is what we need, and SSE works through every corporate proxy without extra infrastructure. Reverb is a Laravel-version-pinned heavyweight dependency we don't want.
- Client-side filtering UI beyond a small set of category chips. Power users go to the existing Failed/Recent pages.
- Realtime metrics (job throughput, latency histograms). v1.x doesn't try to be Prometheus.
- Per-user filtering / favorites / saved searches. Out of scope.

## Architecture

**Recording.** A new `ActivityRecorder` listener subscribes to existing Sunset events and translates each into an `ActivityEvent` value object with a monotonic id, `type` string, `occurred_at` unix timestamp, and a flat `payload` array. The recorder writes the JSON-encoded event to a Redis sorted set (score = id) ‚Äî the same sorted set serves both the replay buffer AND the streaming source. We deliberately chose a fast cursor-poll model (server-side, every 5s) over Redis pub/sub: pub/sub semantics differ subtly between phpredis and predis around blocking reads + read timeouts + heartbeat timers, and the fiddliness isn't worth it for a v1.2.0 that ships against three transport libraries. "Stream" here means "the server polls Redis every 5s and forwards new events as SSE frames," not "Redis publishes directly to the connection." Sub-second freshness becomes a v1.2.1 optimization if anyone asks.

Events captured (matching strings used as `type` field, all snake_case):
- `job_failed` ‚Äî from `Admnio\Sunset\Events\JobFailed`
- `job_completed` ‚Äî from `JobCompleted` (lower priority, filtered out of the default view)
- `job_rate_limited` ‚Äî from `JobRateLimited`
- `job_queued` ‚Äî from `JobQueued` (lower priority, filtered out by default)
- `worker_process_restarting` ‚Äî from `WorkerProcessRestarting`
- `unable_to_launch_process` ‚Äî from `UnableToLaunchProcess`
- `long_wait_detected` ‚Äî from `LongWaitDetected`
- `master_supervisor_deployed` ‚Äî from `MasterSupervisorDeployed`

Events explicitly NOT captured (too noisy or duplicative): `JobQueueing` (pre-flight), `JobReserved`, `JobReleased`, `MasterSupervisorLooped`, `SupervisorLooped`. These fire every loop tick and would drown the stream.

**ID generation.** Each event gets an id from `INCR sunset:activity:next_id`. Strictly monotonic across all workers, supports SSE `Last-Event-ID` resume.

**Replay buffer.** `sunset:activity:stream` ‚Äî sorted set capped to 5000 entries via `ZREMRANGEBYRANK ... 0 -5001` after each `ZADD`. Members are `"{id}:{json}"` to keep entries unique. TTL `86400` (24h) per entry as belt-and-braces against stuck keys.

**Streaming source.** The SSE endpoint cursor-polls the sorted set every `poll_interval_seconds` (default 5) using `ZRANGEBYSCORE ... ($lastEventId +inf LIMIT 0 100`. New events flush to the client as SSE frames. No pub/sub channel ‚Äî see the rationale in Recording above.

**HTTP endpoints** (under existing `/sunset` prefix + Authorize middleware):

- `GET /sunset/activity` ‚Äî Inertia page render. Initial props include the most recent 200 events from the sorted set (descending by id). The Vue page mounts an `EventSource` immediately after first render.
- `GET /sunset/activity/stream` ‚Äî SSE response. Sets `Content-Type: text/event-stream`, `Cache-Control: no-cache, no-transform`, `X-Accel-Buffering: no`, `Connection: close`. Honors the request's `Last-Event-ID` header by replaying from the sorted set first (`ZRANGEBYSCORE` from `last_event_id + 1` to `+inf`) then subscribing to the pub/sub channel. Emits a `:heartbeat\n\n` comment every 15s to keep proxies/firewalls from killing the connection. Closes after `sunset.activity.max_connection_seconds` (default 60); the client reconnects automatically (EventSource does this natively).

**Frontend** ‚Äî new `Activity.vue`:
- Subscribes to `EventSource('/sunset/activity/stream')` on mount. Auto-reconnect handled by EventSource itself (with `Last-Event-ID`).
- Renders a reverse-chronological event log: timestamp, type pill (color-coded by category), one-line summary, expandable JSON payload on click.
- Filter chips: `All` / `Errors` (failed + rate-limited + unable-to-launch) / `Lifecycle` (queued + completed) / `Supervisor` (restarting + deployed + long-wait). Default: `Errors`.
- Pause/Resume button. When paused, EventSource is closed; new-event count badge accumulates so the user sees what they missed.
- Client-side ring buffer of 1000 events. Older entries scroll off; clicking "load older" pages backwards via a `?before_id=` query on the page route (not the stream).

**Failure semantics.**
- Recorder swallows Redis exceptions silently and debug-logs. Telemetry is observability, not load-bearing ‚Äî a Redis outage during job processing must not propagate.
- Stream endpoint catches Redis exceptions, sends a final `event: error\ndata: {"reason":"..."}\n\n`, and closes. Client falls back to polling `?since_id=` every 5 seconds until the stream comes back.
- The "polling fallback" client behavior is deliberately part of the design ‚Äî SSE-incompatible environments (some corporate proxies that strip `text/event-stream`, certain CDN configs) need a working dashboard.

**Public API surface added (committed for v1.x):**
- `Admnio\Sunset\Activity\ActivityEvent` ‚Äî readonly value object
- `Admnio\Sunset\Contracts\ActivityRepository` ‚Äî read contract (`recent(int $limit = 200): array`, `since(int $afterId, int $limit = 1000): array`, `before(int $beforeId, int $limit = 200): array`)
- `Admnio\Sunset\Events\ActivityRecorded` ‚Äî event fired after each successful record, payload is the `ActivityEvent`. Consumers can subscribe to forward to their own audit log / Slack / etc.

**Config** (`sunset.activity.*`):
- `enabled` (bool, default true) ‚Äî when false, the recorder doesn't subscribe and the stream endpoint returns 404.
- `stream_buffer_size` (int, default 5000) ‚Äî replay-buffer cap.
- `max_connection_seconds` (int, default 60) ‚Äî server-side SSE timeout.
- `heartbeat_interval_seconds` (int, default 15) ‚Äî SSE keep-alive cadence.
- `poll_interval_seconds` (int, default 5) ‚Äî server-side sorted-set poll cadence between SSE flushes / heartbeat checks.

## Octane caveat

The SSE endpoint blocks a worker for up to `max_connection_seconds`. With Laravel Octane (Swoole/RoadRunner), this consumes one of your worker slots for that entire duration per connected dashboard tab.

Two options for Octane deployments:
1. **Run the dashboard tier in FPM** behind a separate route ‚Äî Octane handles the rest of the app, FPM serves `/sunset/*`. Documented in README.
2. **Disable streaming** with `SUNSET_ACTIVITY_ENABLED=false` and accept polling. The dashboard's existing 3s polling stays in place.

The README's Octane section gets updated to cover both options.

## Cross-platform notes

- Redis pub/sub works identically on Windows.
- SSE works identically in all modern browsers.
- PHP's `flush()` + chunked output works on every server. We document `output_buffering=0` in php.ini or `ob_end_flush()` in the handler.
- The Windows-compat banner already established for the supervisor process tree applies here too: no new platform-specific concerns introduced.

## Testing strategy

- **Unit:** Factory translates each event into the expected `ActivityEvent` shape. Recorder swallows Redis-down. Repository fails gracefully. Streamer respects max_connection_seconds + heartbeat interval (with injected clock + sleep stubs).
- **Integration:** Recorder e2e ‚Äî fire a `JobFailed` event, assert sorted-set contains an entry with the right type/payload. Controller `show()` returns the right props shape. Stream contract: a basic SSE smoke test that the response headers are right and at least one event flushes through. (Full duplex streaming testing is hard in PHPUnit; skip it. The unit test on `ActivityStreamer` with injected dependencies is the load-bearing test.)
- **No browser test.** The Vue rendering of the event log + EventSource subscription is the kind of thing where a Dusk test pays for itself only if behavior is hard to verify by inspection. Skip for v1.2.0; document a manual smoke-test checklist in the maintainer release notes.
